## Инструменты, которые помогают (помимо SQL)
Прежде чем переходить к исправлениям на SQL, знайте: существуют инструменты, которые облегчат вашу работу:

| Инструмент | Что делает | Когда использовать |
|------------|------------|-------------------|
| dbt | Автоматически запускает тесты SQL, отслеживает линейдж данных | Ежедневные проверки пайплайнов, валидация схемы |
| Great Expectations | Готовые правила качества (проверки на null, диапазоны и т.д.) | Комплексные наборы валидации, документация |
| Airflow/Prefect | Планирует ваши проверки на SQL, отправляет оповещения | Оркестрация проверок качества в пайплайнах |
| BI-инструменты (Tableau, Looker, Metabase) | Визуальные дашборды, показывающие здоровье данных | Видимость для бизнес-стейкхолдеров |
| Оповещения в Slack/Email | Уведомления при сбоях проверок | Немедленная реакция на проблемы |

**Итог:** SQL — это ваша логика валидации. Эти инструменты планируют, запускают и оповещают о результатах этой логики.

## Часть 1: Входящие данные «грязные»
Исправляйте или помечайте проблемы до любой трансформации.

### Дрейф схемы
**Проблема**  
Структура таблицы меняется без предупреждения — столбцы исчезают или появляются новые.

**Что происходит**  
Ваш запрос ломается, потому что выбирает столбец, которого больше не существует. Или вы упускаете критически важные новые данные.

**Пример SQL и исправление**
```sql
-- Проверка текущей схемы перед запуском трансформаций
SELECT column_name, data_type 
FROM information_schema.columns 
WHERE table_name = 'your_table'
ORDER BY ordinal_position;

-- Исправление: Используйте явные списки столбцов, никогда не SELECT *
-- Исправление: Добавьте валидацию схемы в ваш пайплайн
```

### Аномалии объёма
**Проблема**  
Количество строк меняется неожиданно — резкие падения или скачки.

**Что происходит**  
Падение на 50% = сломанный upstream-пайплайн. Скачок на 300% = дублирование данных. Ноль строк = тихий сбой.

**Пример SQL и исправление**
```sql
-- Проверка объёма сегодня против вчера
SELECT 
    COUNT(*) as today_count,
    LAG(COUNT(*)) OVER (ORDER BY CURRENT_DATE) as yesterday_count,
    COUNT(*) * 1.0 / NULLIF(LAG(COUNT(*)) OVER (ORDER BY CURRENT_DATE), 0) as ratio
FROM orders
WHERE order_date = CURRENT_DATE;

-- Исправление: Остановите пайплайн, если количество отличается >20% от среднего
```

### Актуальность данных
**Проблема**  
Пайплайн доставляет устаревшие данные.

**Что происходит**  
Решения принимаются на основе вчерашних данных. Упущенные возможности. Неправильные действия.

**Пример SQL и исправление**
```sql
-- Проверка, насколько стары данные
SELECT 
    MAX(updated_at) as last_update,
    EXTRACT(EPOCH FROM (CURRENT_TIMESTAMP - MAX(updated_at)))/3600 as hours_old
FROM your_table;

-- Исправление: Оповестите, если hours_old > ваш SLA (например, 4 часа)
```

### Дубликаты записей
**Проблема**  
Одна и та же сущность появляется несколько раз.

**Что происходит**  
Выручка учитывается дважды. Количество клиентов завышено. Метрики теряют смысл.

**Пример SQL и исправление**
```sql
-- Поиск дубликатов первичных ключей
SELECT id, COUNT(*) 
FROM your_table 
GROUP BY id 
HAVING COUNT(*) > 1;

-- Исправление: Удалите дубликаты перед агрегацией
-- Селект *, правда, позьмет еще и поле rn. Поэтому не забывайте про совет в начале избегать выбора через *, всегда прописывайте нужные поля!
SELECT * FROM (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY id ORDER BY updated_at DESC) as rn
    FROM your_table
) WHERE rn = 1;
```

### Орфанные записи
**Проблема**  
Внешние ключи ссылаются на несуществующие родительские записи.

**Что происходит**  
JOIN'ы тихо отбрасывают данные. Отчёты занижают показатели. Связи кажутся сломанными.

**Пример SQL и исправление**
```sql
-- Поиск орфанов (заказы без клиентов)
SELECT COUNT(*) as orphan_count
FROM orders o
LEFT JOIN customers c ON o.customer_id = c.customer_id
WHERE c.customer_id IS NULL 
  AND o.customer_id IS NOT NULL;

-- Исправление: Отфильтруйте орфанов или поместите их на карантин для проверки
```

### Несоответствие типов
**Проблема**  
Данные поступают в неправильном типе (текст вместо чисел, даты как строки).

**Что происходит**  
Приведение типов завершается с ошибкой или даёт неверные результаты. Сортировка становится алфавитной вместо числовой.

**Пример SQL и исправление**
```sql
-- Валидация перед приведением типа
SELECT 
    CASE 
        WHEN amount ~ '^[0-9]+\.?[0-9]*$' THEN amount::DECIMAL
        ELSE NULL
    END as amount_clean
FROM orders;

-- Исправление: Очищайте некорректные форматы перед приведением типа
```

### Проблемы с кодировкой и символами
**Проблема**  
Специальные символы, невидимые пробелы или несоответствия кодировки.

**Что происходит**  
'Paris' ≠ 'Paris' (с невидимым символом). GROUP BY создаёт несколько версий одного и того же значения.

**Пример SQL и исправление**
```sql
-- Стандартизация текста
SELECT 
    LOWER(TRIM(REGEXP_REPLACE(customer_name, '[^a-zA-Z0-9\s]', '', 'g'))) as clean_name
FROM customers;

-- Исправление: Всегда очищайте текст при приёме данных
```

### Пробелы и отступы
**Проблема**  
Ведущие/завершающие пробелы или множественные внутренние пробелы.

**Что происходит**  
'ABC Corp' и 'ABC Corp ' не совпадают. JOIN'ы тихо не срабатывают.

**Пример SQL и исправление**
```sql
-- Поиск значений с отступами
SELECT product_code
FROM products
WHERE product_code != TRIM(product_code);

-- Исправление: Всегда используйте TRIM() для строковых полей
SELECT TRIM(BOTH ' ' FROM product_code) as clean_code FROM products;
```

### Стандартизация регистра
**Проблема**  
Одно и то же значение в разном регистре (USA, usa, Usa).

**Что происходит**  
GROUP BY разделяет одну категорию на несколько строк.

**Пример SQL и исправление**
```sql
-- Проверка вариаций регистра
SELECT country, COUNT(*) 
FROM customers 
GROUP BY country 
ORDER BY LOWER(country);

-- Исправление: Стандартизируйте регистр
SELECT LOWER(country) as standard_country FROM customers;
```

### Гибкая категоризация
**Проблема**  
Жёстко закодированные CASE-выражения ломаются при появлении новых значений.

**Что происходит**  
Неклассифицированные значения становятся NULL. Отчёты показывают неполные данные.

**Пример SQL и исправление**
```sql
-- НЕПРАВИЛЬНО: Тихие NULL для новых значений
SELECT 
    CASE region
        WHEN 'US' THEN 'North America'
        WHEN 'CA' THEN 'North America'
    END as region_group
FROM staging.customers;

-- ПРАВИЛЬНО: Видимые несопоставленные значения
SELECT 
    CASE 
        WHEN region IN ('US', 'CA') THEN 'North America'
        WHEN region IN ('UK', 'DE') THEN 'Europe'
        ELSE 'UNMAPPED: ' || region  -- Показывает, что нужно исправить
    END as region_group
FROM staging.customers;
```

### Защитное маппинг значений
**Проблема**  
Таблицы маппинга пропускают целевые значения, создавая пробелы.

**Что происходит**  
'New York' в источнике становится NULL, потому что маппер обрабатывал только 'ny' и 'nyc'.

**Пример SQL и исправление**
```sql
-- НЕПРАВИЛЬНО: Целевое значение не в списке
SELECT 
    CASE 
        WHEN city = 'ny' THEN 'New York'
        WHEN city = 'nyc' THEN 'New York'
        -- 'New York' проваливается в NULL!
    END as city_standardized
FROM staging.locations;

-- ПРАВИЛЬНО: Включите целевое написание
SELECT 
    CASE 
        WHEN city IN ('ny', 'nyc', 'new york', 'New York') THEN 'New York'
        ELSE 'UNMAPPED: ' || city
    END as city_standardized
FROM staging.locations;
```

### Невозможные значения
**Проблема**  
Данные нарушают бизнес-реальность (будущие даты, отрицательный возраст).

**Что происходит**  
Анализ становится ненадёжным. Руководство теряет доверие к данным.

**Пример SQL и исправление**
```sql
-- Пометка невозможных значений
SELECT 
    CASE 
        WHEN order_date > CURRENT_DATE THEN 'FUTURE'
        WHEN order_date < '2000-01-01' THEN 'TOO_OLD'
        WHEN amount < 0 THEN 'NEGATIVE'
        WHEN email NOT LIKE '%@%.%' THEN 'INVALID_EMAIL'
        ELSE 'OK'
    END as flag,
    COUNT(*)
FROM orders
GROUP BY flag
HAVING flag != 'OK';

-- Исправление: Отклоняйте или помещайте на карантин помеченные записи
```

### Выбросы
**Проблема**  
Значения статистически допустимы, но бизнес-невозможны (например, заказ на $1 млн, когда максимум $10 тыс.).

**Что происходит**  
Средние значения искажаются. Агрегации вводят в заблуждение. Мошенничество или ошибки скрыты.

**Пример SQL и исправление**
```sql
-- Пометка статистических выбросов (за пределами 3 стандартных отклонений)
WITH stats AS (
    SELECT AVG(order_amount) as mean_val, STDDEV(order_amount) as std_val
    FROM orders
)
SELECT 
    o.order_id,
    o.order_amount,
    CASE 
        WHEN ABS(o.order_amount - s.mean_val) > 3 * s.std_val 
        THEN 'OUTLIER' ELSE 'OK'
    END as flag
FROM orders o
CROSS JOIN stats s;

-- Исправление: Помечайте, но не удаляйте — пусть бизнес решает
```

### Отсутствующие значения (NULL)
**Проблема**  
Данные отсутствуют.

**Что происходит**  
Агрегации неверны. JOIN'ы отбрасывают строки. Невозможно отличить «ноль» от «неизвестно».

**Пример SQL и исправление**
```sql
-- НЕПРАВИЛЬНО: Тихое преобразование в ноль
SELECT COALESCE(sales, 0) as sales FROM table;

-- ПРАВИЛЬНО: Сохраняйте числовой вывод с флагом
SELECT 
    COALESCE(sales, 0) as sales_value,
    CASE WHEN sales IS NULL THEN 1 ELSE 0 END as is_imputed,
    CASE WHEN sales IS NULL THEN 'MISSING' ELSE 'OK' END as quality_flag
FROM table;
```

## Часть 2: Корректировка данных для анализа
Предотвращайте ошибки в собственных трансформациях.

### Деление на ноль
**Проблема**  
Знаменатель в вычислении равен нулю.

**Что происходит**  
Запрос падает. Дашборд показывает ошибку. Отчёт для руководства не формируется.

**Пример SQL и исправление**
```sql
-- НЕПРАВИЛЬНО
SELECT revenue / orders as aov FROM metrics;

-- ПРАВИЛЬНО
SELECT revenue / NULLIF(orders, 0) as aov FROM metrics;
-- Или: CASE WHEN orders = 0 THEN NULL ELSE revenue / orders END
```

### Ошибки агрегации
**Проблема**  
SUM, AVG, COUNT на «грязных» данных дают неверные результаты.

**Что происходит**  
Двойной учёт из-за дубликатов. Средние значения искажены выбросами.

**Пример SQL и исправление**
```sql
-- Проверка на дубликаты перед агрегацией
SELECT customer_id, order_date, amount, COUNT(*) as occurrences
FROM orders
GROUP BY customer_id, order_date, amount
HAVING COUNT(*) > 1;

-- Исправление: Используйте COUNT(DISTINCT) для уникальных подсчётов
-- Исправление: Очищайте данные перед агрегацией
```

### Ошибки арифметики дат
**Проблема**  
Расчёт интервалов, возрастов, длительностей с NULL или некорректными датами.

**Что происходит**  
Отрицательные возрасты. Деление на ноль в расчётах длительности. Неверный когортный анализ.

**Пример SQL и исправление**
```sql
-- Безопасный расчёт возраста
SELECT 
    customer_id,
    birth_date,
    CASE 
        WHEN birth_date IS NULL THEN NULL
        WHEN birth_date > CURRENT_DATE THEN NULL
        ELSE DATE_PART('year', AGE(CURRENT_DATE, birth_date))
    END as age_years,
    CASE 
        WHEN birth_date IS NULL THEN 'MISSING_BIRTH_DATE'
        WHEN birth_date > CURRENT_DATE THEN 'FUTURE_DATE'
        ELSE 'OK'
    END as flag
FROM customers;
```

### Сбои неявного приведения типов
**Проблема**  
База данных неявно приводит типы, давая неожиданные результаты.

**Что происходит**  
'2024-01-01' > '2024-1-1' (сравнение строк). 5 / 2 = 2 (целочисленное деление).

**Пример SQL и исправление**
```sql
-- НЕПРАВИЛЬНО: Неявное сравнение строк
SELECT * FROM orders WHERE order_date > '2024-01-01';

-- ПРАВИЛЬНО: Убедитесь, что столбец имеет тип DATE
-- Приводите типы только в финальном выводе, не в фильтрах
SELECT 
    order_id,
    amount::DECIMAL(12,2) as amount_decimal
FROM orders
WHERE order_date > '2024-01-01';
```

### Циклические ссылки
**Проблема**  
Само-ссылающиеся вычисления или рекурсивные CTE, которые никогда не завершаются.

**Что происходит**  
Запрос зависает. Ресурсы исчерпаны. Блокировки базы данных.

**Пример SQL и исправление**
```sql
-- Безопасный рекурсивный CTE с лимитом и обнаружением циклов
WITH RECURSIVE hierarchy AS (
    SELECT employee_id, manager_id, 1 as level,
           ARRAY[employee_id] as visited
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    SELECT e.employee_id, e.manager_id, h.level + 1,
           h.visited || e.employee_id
    FROM employees e
    JOIN hierarchy h ON e.manager_id = h.employee_id
    WHERE h.level < 10  -- Предотвращение глубокой рекурсии
      AND NOT e.employee_id = ANY(h.visited)  -- Предотвращение циклов
)
SELECT * FROM hierarchy;
```

### Сверка после трансформации
**Проблема**  
Трансформация неожиданно меняет итоговые суммы.

**Что происходит**  
Источник говорит о выручке $1 млн, дашборд показывает $900 тыс. $100 тыс. исчезают. Никто не доверяет данным.

**Пример SQL и исправление**
```sql
-- Сверка исходных и трансформированных итогов
WITH source AS (SELECT SUM(amount) as total FROM raw_table),
     transformed AS (SELECT SUM(amount) as total FROM clean_table)
SELECT 
    s.total = t.total as match,
    ABS(s.total - t.total) as difference,
    ABS(s.total - t.total) / NULLIF(s.total, 0) * 100 as pct_variance
FROM source s, transformed t;

-- Исправление: Исследуйте, если отклонение > 0,1%
```

## Часть 3: Мониторинг и оповещение
Систематическое наблюдение за здоровьем данных.

### Архитектура DWH для качества
**Трёхуровневая архитектура качества:**

| Слой | Назначение | Механизм качества |
|------|------------|-------------------|
| Staging | Приём сырых данных | Валидация схемы, проверка типов, карантин плохих записей |
| Intermediate | Очищенные/трансформированные данные | Целостность ссылок, бизнес-правила, сверка |
| Mart | Готовые для бизнеса данные | Валидация метрик, семантические проверки, обнаружение дрейфа |

**Паттерн карантина:** Отделяйте некорректные записи от корректных, позволяя пайплайну продолжать работу, изолируя проблемы.

### Методы мониторинга
1. **Ограничения таблиц (жёсткое обеспечение)**
```sql
-- Принудительное применение правил на уровне базы данных
ALTER TABLE orders ADD CONSTRAINT chk_positive_amount CHECK (amount >= 0);
ALTER TABLE customers ADD PRIMARY KEY (customer_id);
```

2. **Тесты dbt (мягкое обеспечение)**
```yaml
# Запускайте автоматически в вашем пайплайне
columns:
  - name: order_id
    tests:
      - unique
      - not_null
  - name: amount
    tests:
      - dbt_utils.accepted_range:
          min_value: 0
```

3. **Отдельные таблицы QA (аудиторский след)**
```sql
-- Логирование метрик качества во времени
CREATE TABLE audit.daily_quality_metrics (
    check_date DATE,
    table_name VARCHAR(100),
    check_name VARCHAR(100),
    metric_value NUMERIC,
    status VARCHAR(20)
);
```

4. **Мониторинг через BI-дашборды**
- Оценка здоровья по таблице (0–100%)
- Трендовые линии, показывающие качество во времени
- Топ часто проваливающихся проверок

### Каналы оповещения

| Приоритет | Канал | Время реакции | Примеры |
